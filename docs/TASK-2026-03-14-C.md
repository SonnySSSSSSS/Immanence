# TASK-2026-03-14-C: Make Day 1 Actionable Immediately After Beginning Initiation

**Created:** 2026-03-14
**Recommended Executor:** Codex
**Scope:** Initiation contract start-date anchoring only (single atomic behavior fix)
**Flags:** `IMPLEMENT UNCOMMITTED, VERIFY FIRST` `NO COMMIT WITHOUT HUMAN APPROVAL`

---

## Goal

Fix the initiation start-date anchoring so that after a new student completes setup and clicks `Begin Contract`, the first scheduled practice can become actionable in the current onboarding session when the selected slot has not already passed.

---

## Problem Statement

The initiation setup currently persists a valid active path, but it always anchors Day 1 to the next calendar day.

As a result:
- the user finishes setup successfully,
- the app returns to HomeHub,
- both Day 1 actions often show `Not Yet`,
- the student cannot immediately begin the first practice from the same session,
- the real “sign up -> choose student -> start initiation -> do practice” process breaks right after onboarding.

This was the second concrete blocker from the readiness audit, after the student-mode auth trap.

---

## Wrong Behavior To Fix

**Actual wrong behavior**
- User completes initiation onboarding with valid days, two valid time slots, and a benchmark.
- User clicks `Begin Contract`.
- The app creates an active path successfully.
- Day 1 is anchored to tomorrow even when today’s first selected slot is still in the future.
- HomeHub shows the first leg as `Not Yet`, so the user cannot start immediately.

**Concrete runtime proof from audit**
- Local time during audit: `Sat Mar 14 2026 10:34 GMT+0700`
- Selected first slot: `11:03`
- After `Begin Contract`, HomeHub showed:
  - `1 Mar 15 Morning Breath Practice 11:03 Not Yet`
- Persisted active path stored:
  - `startedAt: 2026-03-15T03:32:00.000Z`

**Exact code boundary**
- Start-date anchor logic: `src/utils/scheduleUtils.js`
- Callers that consume the anchor:
  - `src/state/navigationStore.js`
  - `src/state/curriculumStore.js`

---

## Expected Right Behavior

When a student begins the initiation contract:
- if the first selected slot is still upcoming today, Day 1 should anchor to today at that slot,
- if that slot has already passed beyond the allowed late window, Day 1 should anchor to the next valid day,
- the post-onboarding HomeHub should show an actionable first practice when the chosen slot is still legitimately ahead,
- the user should be able to move directly from onboarding into practice without waiting until tomorrow unless today’s slot is actually no longer valid.

---

## Real Process Problem This Fix Addresses

This fix addresses a real flow break in the student onboarding process:
- a student can now authenticate and choose `Student`,
- but after completing setup they still cannot start the actual initiation practice in the same session,
- the process feels false-complete because setup succeeds while the first real action stays unavailable,
- this creates confusion at the exact handoff from onboarding to doing the practice.

In practical terms, the onboarding promises a start, but the current anchor logic silently delays it by one day.

---

## Single Hypothesis

The immediate-actionability failure exists because `computeScheduleAnchorStartAt()` currently always returns the next calendar day for the first slot, instead of checking whether today’s first slot is still within a valid future/active start window.

---

## Single Atomic Change

Change the contract start-date anchor logic so it uses today’s first slot when that slot is still actionable, and only rolls to the next day when today’s first slot has already expired beyond the configured start window.

No unrelated onboarding, practice, auth, or persistence refactors.
