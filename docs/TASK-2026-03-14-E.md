# TASK-2026-03-14-E: Sync Active Path Progress After Completed Scheduled Practice

**Created:** 2026-03-14
**Recommended Executor:** Codex
**Scope:** One atomic state-sync fix for active path progress facts after a completed scheduled practice
**Flags:** `IMPLEMENT UNCOMMITTED, VERIFY FIRST` `NO COMMIT WITHOUT HUMAN APPROVAL`

---

## Goal

Make a completed scheduled path practice immediately update the active path's persisted progress snapshot:

`scheduled practice complete -> record session -> activePath.progress reflects the completion`

---

## Problem Statement

The student practice chain now works through launch, `SESSION COMPLETE`, and persisted slot completion on the HomeHub card.

But one state surface still lags behind:

- `immanenceOS.progress` records the completed session correctly
- `immanenceOS.navigationState.activePath.progress` still stays at zeroed facts such as:
  - `sessionsCompleted: 0`
  - `totalMinutes: 0`
  - `daysPracticed: 0`
  - `lastSessionAt: null`

This leaves the path run's own persisted progress snapshot out of sync with the actual completed practice history.

---

## Wrong Behavior To Fix

**Current wrong behavior**

- A student completes a scheduled path practice successfully
- the app shows `SESSION COMPLETE`
- the HomeHub slot can show `Done`
- but the active path's persisted progress snapshot still says no sessions were completed

**What this task is fixing**

- The missing state sync between recorded path sessions and `navigationState.activePath.progress`

**Exact real process problem**

- The student has actually practiced, but path reports, status panels, and any logic reading `activePath.progress` can still display zero progress.
- This creates contradictory state inside the same student flow and weakens trust in whether the app really recorded the practice.

---

## Expected Right Behavior

After a scheduled path session completes:

- `immanenceOS.progress.sessionsV2` contains the completed session
- `immanenceOS.navigationState.activePath.progress.sessionsCompleted` increments
- `immanenceOS.navigationState.activePath.progress.totalMinutes` increases
- `immanenceOS.navigationState.activePath.progress.daysPracticed` and streak fields remain aligned with strict obligation-day completion semantics
- `immanenceOS.navigationState.activePath.progress.lastSessionAt` is set

No duplicate counting and no divergence between progress stores.

---

## Real Process Problem This Fix Addresses

This fixes the remaining trust gap inside the practice process:

- the student can practice,
- the completion is recorded,
- and the path itself now acknowledges that completed practice in its own persisted state.

Without this, the app still contains two conflicting answers to the question "did the student make progress on this path?"

---

## Single Hypothesis

If the active path progress snapshot stays stale after completed scheduled practice, the missing update is at the session-recording-to-navigation-state sync boundary.

---
