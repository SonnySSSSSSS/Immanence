# TASK-2026-03-14-B: Restore Student Mode Selection After Auth

**Created:** 2026-03-14
**Recommended Executor:** Codex
**Scope:** Auth -> user-mode handshake only (single atomic behavior fix)
**Flags:** `IMPLEMENT UNCOMMITTED, VERIFY FIRST` `NO COMMIT WITHOUT HUMAN APPROVAL`

---

## Goal

Fix the first-time student entry blocker so an authenticated user can choose `Student` once, persist that choice to their account-scoped mode state, and proceed into the student flow instead of getting trapped at the chooser.

---

## Problem Statement

The app now renders a `Student / Explorer` chooser after auth, but the authenticated user ID is not being bound into `userModeStore` before `setUserMode()` runs.

As a result:
- the user can click `Student`,
- the chooser remains on screen,
- `hasChosenUserMode` stays false,
- no per-user mode is written,
- the student onboarding flow cannot begin.

This is the highest-severity blocker found in the audit because it stops the exact student process before path setup, practice launch, or completion can even be tested from the real flow.

---

## Wrong Behavior To Fix

**Actual wrong behavior**
- Signed-in user sees the `Student / Explorer` chooser.
- User clicks `Student`.
- The chooser does not dismiss.
- HomeHub student flow does not unlock.
- Local persisted mode remains empty because `activeUserId` is missing.

**Concrete runtime proof from audit**
- With a valid auth session, clicking `Student` left the chooser visible.
- Persisted storage remained:
- `{"state":{"modeByUserId":{}},"version":2}`

**Exact code boundary**
- Chooser render and click target: `src/App.jsx`
- Per-user mode write guard: `src/state/userModeStore.js`
- Auth event source: `src/components/auth/AuthGate.jsx`

---

## Expected Right Behavior

When an authenticated user reaches the chooser:
- the app resolves the current auth user ID into `userModeStore`,
- clicking `Student` writes `student` for that user ID,
- `hasChosenUserMode` becomes true,
- the chooser dismisses immediately,
- the student lands in the correct student HomeHub flow,
- reload restores the same user-specific mode without asking again.

---

## Real Process Problem This Fix Addresses

This fix addresses a real onboarding failure in the student process:
- a new student can currently create or sign into an account but cannot actually enter the student product path,
- the initiation setup flow is blocked before it starts,
- any audit of practice completion from the real student path becomes invalid because the user never clears the first gate,
- reloads make the problem look worse because the user appears authenticated but still cannot progress.

In practical terms, this is not cosmetic. It prevents students from starting to use the app at all.

---

## Single Hypothesis

The student-mode trap exists because the app handles auth session changes in `App.jsx` but never calls `useUserModeStore.setActiveUserId(authUser.id)`, so `setUserMode()` runs without an active user binding and intentionally refuses to mark the choice as chosen.

---

## Single Atomic Change

Wire the authenticated user ID into `userModeStore` during auth session lifecycle handling so the existing chooser can persist `student` or `explorer` per user and dismiss correctly.

No unrelated onboarding, navigation, or schedule behavior changes.
