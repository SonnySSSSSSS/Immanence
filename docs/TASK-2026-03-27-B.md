## Goal

The `progressStore` ↔ `navigationStore` ↔ `curriculumStore` pipeline is the most complex stateful logic in the codebase. It enforces schedule anchoring, curriculum precision rail matching, and path contract validation. Changes to any of the three stores can silently break invariants in the others. Currently, this pipeline has zero automated test coverage — only Playwright smoke tests, which do not exercise store logic directly.

This task adds:
1. Vitest as a dev dependency and minimal config.
2. A focused test file covering the key sync invariants for this pipeline.

---

## Problem Statement

The three-way sync has documented invariants in `docs/ARCHITECTURE.md` (schedule anchor computation, precision rail thresholds, path contract validation). Any store schema change or migration risks breaking these invariants without an immediate signal. The only current signal is a failed Playwright smoke test or a user-visible regression.

Adding unit tests here gives:
- Regression protection during store migrations
- A concrete spec of the sync contract that executing agents can reference
- Fast local feedback (`vitest run`) without needing a browser

---

## Single Bounded Change

Add Vitest dev dependency, minimal config, and one test file covering the progress↔navigation↔curriculum sync invariants. No changes to production store files.

## Reuse-First Plan (Mandatory)

### Step 1 — Add test script

**Note (2026-03-27):** Vitest 2.x is not compatible with rolldown-vite 7.2.5. The SSR transform rolldown generates references `__vite_ssr_exportName__` which Vitest's module runner does not provide at runtime. This is a known rolldown-vite/Vitest integration issue. The project already has a working test infrastructure using Node's native `node:test` runner in the `test/` directory; the new tests use that infrastructure instead.

`package.json`:
- Add a `"test"` script: `"test": "node --test test/storeSyncInvariants.test.js"`.

### Step 2 — Write test file

Create `src/state/__tests__/storeSyncInvariants.test.js`.

The test file must cover:

1. **Schedule anchor invariant** — Day 1 is determined by the path activation date anchor, not the session recording time. Verify that `navigationStore`'s schedule computation does not shift the anchor when a session is recorded on a non-day-1 date.

2. **Curriculum precision rail** — A session within ≤15s of target duration should be rated green; a session within ≤60s should be rated red; beyond 60s should be out-of-tolerance. Verify the threshold logic from `curriculumStore` or its supporting utils.

3. **Path contract validation** — A path with `totalDays` set must reject a slot schedule that exceeds `practicePerWeek`. Verify the validation predicate.

4. **User isolation** — State written for `ownerUserId: "user-a"` must not be readable by a store reset to `ownerUserId: "user-b"`.

For each test, import the store or its helper/util directly. Use Vitest's `beforeEach`/`afterEach` to reset store state between tests. Do not mock the stores — test them directly.

If a test requires a helper or utility that is not directly importable (e.g., it is embedded inside a component), extract the relevant predicate into a comment explaining why it cannot be tested directly, and mark the test as `.todo`.
