### A2 — Hook called after early return (`PathOverviewPanel.jsx:61` + unused var at `:225`)

**Hook violation:** `if (!path || path.placeholder) return null;` at line 61 appears before a `useEffect` at line 208. When `path` is null or a placeholder, the early return skips the effect — conditionally calling a hook. This violates the Rules of Hooks.

**Unused variable:** `ensureViewedPathLoaded` is declared at line 225 but never called. ESLint `no-unused-vars` fires.

**Fix (hook violation):**
1. Remove the early return from line 61.
2. Wrap the `buildPathOverviewViewModel(...)` call and all downstream variable declarations that depend on a non-null `path` in a single conditional expression or guard block:
   ```js
   const viewModel = (path && !path.placeholder)
     ? buildPathOverviewViewModel({ path, activePath, resumablePathId, ... })
     : null;
   const {
     isInitiationPath, isAcceptancePath, isViewedPathActive, isViewedPathResumable,
     contract, orderedDayOptions, selectedDays, scheduleTimes, scheduleConstraint,
     scheduleValidation, requiredDays, daysValidation, benchmarkValidation,
     canBeginPath, totalSteps, canAdvanceCurrentStep, scheduleInstruction
   } = viewModel ?? {};
   ```
3. Move `if (!path || path.placeholder) return null;` to just before the final `return (` JSX statement.

This pattern ensures all hooks are called unconditionally on every render while path-dependent logic still bails out before rendering.

**Fix (unused var):** Delete the `ensureViewedPathLoaded` function declaration at line 225 (lines 225–234). Verify it is not called anywhere in this file before removing.

### A4 — Ref mutation during render (`AvatarV3.jsx:112`)

Lines 106–114 conditionally assign `avatarV3ProbeIdRef.current` directly in the render function body:
```js
if (avatarV3ProbeIdRef.current == null) {
  if (AVATAR_V3_HMR_HOST_PROBE_ENABLED) {
    ...
    avatarV3ProbeIdRef.current = probe.avatarV3MountSeq;
  } else {
    avatarV3ProbeIdRef.current = 'host-probe-disabled';  // line 112
  }
}
```

Mutating a ref during render is flagged by the `react-hooks/exhaustive-deps` and `no-render-side-effects` rules. The probe ID is only used in mount/unmount/render-log calls — it is read-once initialization.

**Fix:** Move the initialization block into the existing `useEffect` at line 116 (the mount effect), placing it before the `logAvatarV3HmrHostProbe('mount', ...)` call. The condition `if (avatarV3ProbeIdRef.current == null)` stays intact. The ref will be set synchronously on first mount before any log calls in that effect.

```js
useEffect(() => {
  // Initialize probe ID once on mount
  if (avatarV3ProbeIdRef.current == null) {
    if (AVATAR_V3_HMR_HOST_PROBE_ENABLED) {
      const probe = getAvatarV3HmrHostProbeContext();
      probe.avatarV3MountSeq += 1;
      avatarV3ProbeIdRef.current = probe.avatarV3MountSeq;
    } else {
      avatarV3ProbeIdRef.current = 'host-probe-disabled';
    }
  }
  logAvatarV3HmrHostProbe('mount', { ... });
  return () => { ... };
}, []);
```

The `logAvatarV3HmrHostProbe('render-parent-props', ...)` calls at lines 167 and 177 read `ref.current` during render. This is acceptable — reading a ref during render does not violate React rules and does not trigger lint errors.
