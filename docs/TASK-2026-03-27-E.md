## Goal

`transition: all` causes the browser to watch every CSS property on an element for changes and animate any that change. On elements with complex box-shadows, filters, or backdrop-filters, this triggers GPU compositing work on every state change even when only one property (e.g., opacity, width) needs to animate. The 200+ occurrences across this codebase are not all worth fixing, but the ones below are high-frequency (run during active sessions) or have notably long durations that make the sweep visible.

This task targets the **6 highest-impact files** only. Do not attempt a global `transition-all` sweep.

## Problem Statement

### Target Files and Changes

#### 1. `src/components/CompactStatsCard.jsx`

Lines 116, 140: `transition-all duration-1000`

A 1-second all-properties transition on a stats card and its blur overlay. `duration-1000` with `all` means any CSS change on these elements will animate for a full second. The only property that actually changes is width (progress bar fill) and opacity (glow overlay).

**Replace:**
- `transition-all duration-1000` on the stats bar → `transition-[width] duration-1000`
- `transition-all duration-1000` on the blur overlay → `transition-[width,opacity] duration-1000`

#### 2. `src/components/ApplicationTrackingCard.jsx`

Line 71: `transition-all duration-700 ease-in-out` on the card root element.

700ms all-properties on a card with backdrop-filter and box-shadow. The card root only needs to animate visual state changes (likely opacity or a color shift). Cards with `backdrop-filter: blur()` are expensive to repaint.

Read the component to confirm which CSS properties actually change on this element, then replace `transition-all duration-700 ease-in-out` with the specific properties that change (likely `transition-[opacity,background-color] duration-700 ease-in-out` or similar).

Line 141: `transition-all duration-300` on an inner action button. Buttons typically only need `transition-[color,background-color,border-color,opacity] duration-300`.

#### 3. `src/components/DailyPracticeCard.jsx`

Lines 657, 1808: `transition: 'all 0.7s ease-in-out'` (inline style).

700ms all-properties sweep on what appears to be the main card body. Read the surrounding context on each line to determine which properties actually animate (look at conditional className or style changes on the same element), then narrow the transition to those properties.

## Implementation Rules

1. **Narrow, don't widen.** Replace `transition-all` or `transition: all` with only the properties that actually animate on that element.

2. **Preserve duration and easing exactly.** The timing and curve must be identical — only the property scope changes.
   ```
   transition-all duration-300  →  transition-[color,background-color] duration-300
   transition: 'all 0.7s ease-in-out'  →  transition: 'opacity 0.7s ease-in-out, transform 0.7s ease-in-out'
   ```

3. **Tailwind arbitrary syntax for non-standard properties:**
   ```
   transition-[width,opacity]          ✓ valid Tailwind arbitrary value
   transition-[backdrop-filter]        ✓
   transition-[color,border-color]     ✓
   ```

4. **Inline style syntax for JS objects:**
   ```js
   transition: 'opacity 0.3s ease, background-color 0.3s ease'
   ```

5. **If uncertain, use `transition-[opacity,transform]` as a safe minimum** — these two cover the most common visual animations without triggering layout or paint.
