## Goal

`src/components/PracticeSection.jsx` is 3,682 lines and owns several distinct flows: session configuration, active session, post-session summary, and journaling. This task determines whether the post-session summary — the UI shown after a session completes — is a clean extraction candidate.

The summary is expected to be a terminal presentation state that renders only when a session has ended. If this expectation holds and the boundary is clean, extract it into `PracticeSessionSummary.jsx`. If it does not hold, stop and report what prevents extraction.

This is an audit-then-implement task, not a guaranteed extraction.

---

## Problem Statement

`PracticeSection.jsx` renders the post-session summary inline. The summary surface likely covers:

- Displaying session stats (duration, quality, streak)
- Rendering the session recap card
- Providing navigation actions (back to practice, mark complete)

This logic is embedded in the same 3,682-line file as active session rendering, making it impossible to review or test in isolation.

**Before implementing anything:** the executing agent must locate all render paths and handlers exclusively involved in post-session summary presentation, regardless of whether they are grouped under a single conditional branch. Do not assume the summary is a single contiguous block keyed on one state variable.

## Audit-First, Then Implement If Clean

This task has two phases. Phase 1 (audit) always runs. Phase 2 (extraction) runs only if Phase 1 confirms a clean boundary.

## Phase 1 — Pre-Implementation Audit (Mandatory)

Before writing any code, the executing agent must:

1. Read `src/components/PracticeSection.jsx` and identify:
   - All render paths that constitute summary presentation (start line → end line for each).
   - Whether the summary surface is contiguous in one render block, or split across multiple distant branches.
   - All local state variables, handlers, and refs used exclusively within the summary surface (extraction candidates).
   - All props or store values read by the summary surface that originate outside it (these become props of the new component).
   - All helper functions used by the summary surface — and whether any of those helpers are also used by non-summary flows.

2. Confirm the extraction is clean. All of the following must hold:
   - No variable in the summary surface is also written to by the active session flow.
   - No handler in the summary surface triggers state transitions in the active session flow.
   - The summary surface is contiguous or can be unified without refactoring non-summary logic.
   - No required helper functions are materially shared with non-summary flows (trivial utilities like formatters are acceptable; session-transition logic is not).
   - Extraction requires moving only presentation logic, not session-transition logic.

3. If any of the above conditions fail, **stop and report the specific coupling.** Do not proceed with a partial extraction.

4. Report the identified line range(s), the props/callback inventory, and any shared helpers before implementing.

## Phase 2 — Mechanical Extraction (Only If Phase 1 Passes)

### Step 1 — Create `PracticeSessionSummary.jsx`

Create `src/components/practice/PracticeSessionSummary.jsx`:

- Move the identified JSX and its local state/handlers out of `PracticeSection.jsx` into this file.
- **Preserve current data ownership.** If `PracticeSection.jsx` currently reads from stores and passes derived values into the summary branch, the new component receives those values as props. Do not introduce new store reads in the extracted component.
- Use a named export: `export function PracticeSessionSummary({ ... }) { ... }`.

### Step 2 — Update `PracticeSection.jsx`

In `src/components/PracticeSection.jsx`:

- Import `PracticeSessionSummary` from `./practice/PracticeSessionSummary.jsx`.
- Replace the extracted inline block with `<PracticeSessionSummary ... />` passing the identified props.
- Remove the extracted local state/handlers from `PracticeSection.jsx`.
